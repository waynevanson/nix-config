# Manual WordPress module for multiple instances

Created: 2026-06-18T06:55:09Z

## Goal

Replace the NixOS `services.wordpress` module usage with a custom, reusable NixOS
module that runs mutable WordPress instances from writable directories, making it
easy to add more than one site.

## Context

- Current `hosts/server/wordpress-wayne.nix` uses `services.wordpress` plus
  bind-mount hacks to make `wp-content` mutable.
- `hosts/server/wordpress-lx.nix` is already written for a second site but is
  commented out in `hosts/server/default.nix`.
- The user wants full WordPress admin mutability (plugins, themes, core updates)
  and the ability to host many instances.
- Existing server already runs nginx, ACME (via spaceship for waynevanson.com),
  and MySQL/MariaDB.

## Assumptions

1. WordPress core files live under `/var/lib/wordpress/<domain>/wordpress`.
2. Each instance gets its own MariaDB database and DB user.
3. Each instance gets its own PHP-FPM pool running as a dedicated system user
   `wordpress-<domain>` and the shared `nginx` group. (Confirmed: per-instance
   users.)
4. WordPress core is sourced from `pkgs.wordpress` but copied to the mutable
   directory on first install.
5. `wp-config.php` is generated by Nix and contains DB socket settings and
   sops-managed WordPress salts.
6. SSL uses existing `security.acme` + `services.nginx.virtualHosts` setup.
7. Each instance is declared in a small attrset like:

   ```nix
   services.custom.wordpress.instances = {
     "waynevanson.com" = {
       adminEmail = "...";
       acmeHost = "waynevanson.com";
     };
     "luscomberecords.com" = {
       adminEmail = "...";
       acmeHost = "luscomberecords.com";
     };
   };
   ```

## Tasks

- [ ] **Design the module interface**
  - Decide option names and shape for `services.custom.wordpress.instances`.
  - Document required vs optional fields.

- [ ] **Create the custom module file**
  - Add `modules/custom/services/wordpress.nix`.
  - Implement option declarations for instances.

- [ ] **Create system users and groups per instance**
  - One `wordpress-<domain>` user per instance.
  - All instances share the `nginx` group for webserver access.

- [ ] **Provision MariaDB databases and users**
  - Use `services.mysql.ensureDatabases` and `ensureUsers`.
  - No DB password: use unix socket authentication. DB user matches the Linux
    user `wordpress-<domain>` so MariaDB's `unix_socket` plugin grants access.

- [ ] **Generate wp-config.php per instance**
  - Use `pkgs.writeTextFile` to create `wp-config.php`.
  - Include DB_NAME, DB_USER, DB_PASSWORD = "", DB_HOST = socket path, salts.
  - Define `FS_METHOD = "direct"`, `WP_HOME`, `WP_SITEURL`, `FORCE_SSL_ADMIN`.

- [ ] **Create a one-shot setup service per instance**
  - Copy `pkgs.wordpress` core files into `/var/lib/wordpress/<domain>/wordpress`
    if not already present.
  - Ensure `wp-content/upgrade` exists.
  - Symlink the generated `wp-config.php` into the instance root.
  - Set correct ownership (`wordpress-<domain>:nginx`).

- [ ] **Create PHP-FPM pools per instance**
  - One pool per instance using `services.phpfpm.pools."wordpress-<domain>"`.
  - Configure `user`, `group`, `listen`, PHP options.
  - Set `open_basedir` to the instance root and `/var/lib/wordpress/<domain>`.

- [ ] **Create nginx virtual hosts per instance**
  - Use `services.nginx.virtualHosts.<domain>`.
  - Set `root` to `/var/lib/wordpress/<domain>/wordpress`.
  - Add `serverAliases = [ "www.<domain>" ]` automatically.
  - Replicate standard WordPress nginx rules (try_files, PHP handling, uploads
    PHP block, static expires).
  - Set `useACMEHost` or `enableACME` per instance.

- [ ] **Migrate `wordpress-wayne.nix` to the new module**
  - Remove `services.wordpress`, bind-mount hacks, and custom units.
  - Replace with a declaration in `services.custom.wordpress.instances`.

- [ ] **Migrate `wordpress-lx.nix` to the new module**
  - Uncomment the import in `hosts/server/default.nix`.
  - Convert to the new instance declaration.

- [ ] **Keep wp-cli available on the host**
  - Add `pkgs.wp-cli` to `environment.systemPackages` in the host config or
    inside the module.

- [ ] **Clean up `hosts/server/default.nix`**
  - Remove the commented-out wordpress-lx import if it's now active.
  - Ensure ACME and nginx base config still apply.

- [ ] **Add sops secrets for WordPress salts**
  - Add one secret per instance (or per salt field) to `.sops.secrets.yaml`.
  - Reference them in the module via `config.sops.secrets` and inject into
    `wp-config.php`.

- [ ] **Test the migration**
  - Run `nix flake check`.
  - Deploy and verify both sites load.
  - Verify plugin/theme installation works without FTP prompts.
  - Verify adding a third instance requires only a new attrset entry.

## Decisions

| Question | Decision |
| --- | --- |
| Separate user per instance? | **Yes** — one `wordpress-<domain>` user per instance. |
| DB password management? | **No password** — unix socket authentication via matching DB user. |
| `www.<domain>` aliases? | **Yes**, automatic. |
| Keep `wp-cli` on host? | **Yes**. |
| Fully mutable `wp-content`? | **Yes**, copy full `wp-content` on first install. |
